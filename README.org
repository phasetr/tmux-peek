* tmux-peek

tmux-peek is a small Emacs Lisp library for inspecting live tmux state
asynchronously.

It is a personal tool for GNU Emacs 30.1 and tmux 3.6a.  It is designed
around the needs of [[https://github.com/phasetr/enkan-repl][enkan-repl]], but remains independent.  enkan-repl owns
start/send/attach/mirror style operation flows; tmux-peek owns state
inspection and explicit session cleanup.

** Current Scope

- Query sessions, windows, panes, clients, paste buffers, options, and
  environment.
- Expand tmux formats through =display-message -p=.
- Capture pane text, including one-shot tail-like capture through
  =:tail-lines=.
- Show paste-buffer contents.
- Check whether a tmux server is running.
- Check whether a target exists.
- Return the tmux version string.
- Kill only an explicitly selected session through =kill-session=.

All public command wrappers are asynchronous and call a callback with a
result plist.

#+begin_src elisp
(tmux-peek-list-panes-async
 (lambda (result)
   (when (plist-get result :ok)
     (message "%S" (plist-get result :value))))
 '(:socket-name "main"))
#+end_src

Successful results contain =:ok t= and usually =:value=.  Failed results
contain =:ok nil= and =:error=.

** Explicitly Unsupported

tmux-peek intentionally does not provide wrappers for these tmux operations:

- =kill-pane=
- =send-keys=
- =new-session=
- =new-window=
- =split-window=
- =attach-session=
- =kill-window=
- =kill-server=
- tmux interactive UI commands such as =choose-*=, =display-menu=,
  =display-popup=, and =command-prompt=

Only =kill-session= is exposed for cleanup, because enkan-repl's tmux
lifecycle is session-level.  Window, pane, and server cleanup wrappers are
intentionally absent because they are not part of the current cleanup contract.

** Synchronous Helpers

Synchronous helper functions are not implemented at this stage.

The main reason to add synchronous helpers would be compatibility with call
sites that cannot conveniently accept callbacks.  That is not the current
core path.  Emacs process callbacks, tmux calls, and enkan-repl adjacency make
asynchronous APIs the safer default because they avoid blocking the editor and
avoid building a second execution path around =accept-process-output=.

If synchronous helpers are added later, they should be thin wrappers over the
same async executor, not a separate tmux execution implementation.

** Development

Run unit tests and byte compilation:

#+begin_src sh
make check
#+end_src

Run integration tests against a dedicated tmux socket:

#+begin_src sh
make test-integration
#+end_src

Refactoring is part of the implementation plan.  At phase boundaries, keep the
executor, command builder, parser, and public API small, remove unused
abstractions, and keep enkan-repl-covered operation flows out of this package.
